# 安全策略

English guide: [SECURITY.md](SECURITY.md)

本仓库正在按“正式开源发布”的标准整理。参考 GitHub 对仓库安全策略的常见做法，本文件说明哪些问题属于安全范围，以及应该如何报告。

## 当前重点安全范围

这些方面尤其需要关注：

- provider API Key 的保存和使用方式
- 本地 runtime 更新与切换链路
- 安装器完整性与桌面启动流程
- AIAnalysis 的备份、恢复、导出和本地持久化
- 请求签名、错误输出和日志脱敏
- 桌面桥接下的文件导入 / 导出路径

## 支持版本范围

当前计划支持的范围是：

- 当前默认分支
- 正在维护的 `1.2.x` 工作线
- 仓库正式公开后，最新一条公共发布线

更旧的私有快照、个人 fork 或大幅改造过的本地版本，原则上只提供 best-effort 支持。

## 如何报告漏洞

以下问题**不要**直接发成公开 Issue：

- API Key 泄露
- 安装器绕过或 runtime 篡改
- 本地数据外流
- 日志或导出内容中的凭据暴露
- 可能导致代码执行的安全问题

### 正式公开后的推荐路径

仓库正式公开后，推荐的报告顺序是：

1. GitHub 私密漏洞报告，在仓库设置中启用
2. 单独的安全联系邮箱

当前安全联系邮箱：

- 安全邮箱：`maxwelldhx@gmail.com`

### 当前本地 / 公开前路径

在仓库正式对外开放并配置好安全通道之前：

- 请保持私下报告
- 准备一份已脱敏的本地报告
- 只包含最少但足够复现的信息

建议带上：

- 受影响版本、分支或 runtime 线
- 问题出现在 web、app，还是两端
- 精确复现步骤
- 影响评估
- 已脱敏的日志、截图或备份片段

## 处理预期

- 敏感报告在修复前应保持私下处理
- 在补丁发布前，公开变更说明不应附带过细的利用步骤
- 审计过程中发现的任何密钥或凭据，都应在公开发布前完成轮换或移除
- 若修复影响存储、导出、provider 契约或 runtime 行为，应清楚写进文档
