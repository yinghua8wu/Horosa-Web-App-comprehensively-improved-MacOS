# 更新后重启「卡在 100% 很久才进得去」修复（v2.3.1）

## 现象

用户反馈:Mac 软件**在线更新之后第一次重启**，启动进度条走到 / 停在 **100%**，要等很久才进得了主界面；而且「之前也有这个问题」（每次更新都重演）。平常（非更新）重启则很快。

## 根因（实测 + 代码定位）

对比这台机器 `~/Library/Application Support/com.horacedong.horosa/logs/` 的真实启动计时:

| 场景 | 后端就绪耗时(脚本 run begin→ready) | 走的路径 |
|---|---|---|
| 平常重启 | **6–8 秒** | 快路径 `trusted_runtime=1` / `mongo_skip_ping=1` / fast-path / 跳过 warmup |
| 更新后首启 | **27–29 秒** | 慢路径 `startup_timeout=300` / `mongo_skip_ping=0` / `trusted_runtime=0` / 禁 fast-path / 同步 warmup |

「100%」是启动进度的最后一格 `emit_progress(100, "星阙 已准备完成")`（`Horosa_Desktop_Installer/src-tauri/src/main.rs` `runtime_bootstrap`）。到 100% 之后才 `emit_ready` → `window.location.replace(真实应用URL)` 导航到 React 应用。所以「卡在 100%」= 后端慢路径（进度条停在 82%）+ 前端冷加载（停在 100%）叠加。更新后**两头都是冷的**（.app 整包被替换 → OS 文件缓存冷 + Gatekeeper 重验 + Java/Python 冷启动），所以特别久。

慢路径**本身是设计如此**（更新后要对刚装好的运行时做完整校验），但有 4 个可优化/有缺陷的点:

1. **进度无反馈像死机**：慢路径期间进度条停在固定数字 30–60 秒，无任何「正在校验」提示。
2. **warmup 同步阻塞**：`warm_runtime_routes` 在 services ready 之后、`run end` 之前**同步**跑（2–3s），白白拖长启动。
3. **mongo 启动期 ping**：`mongo_skip_ping=0` 时启动要 ping MongoDB；mongo 是可选服务（`mongo_optional=1`），缺席时 ping 会拖满等待。
4. **轮询粒度粗**：非 trusted 路径 `poll_interval=1s`（trusted 是 0.1s），服务就绪后最多白等近 1 秒。

另有两个**会让它复发 / 更糟的真 bug**:

5. **🐛 更新标记仅「成功时」消费 → 失败残留致次次都慢**：`update-complete.txt` 只在 `runtime_bootstrap` 成功分支经 `show_post_update_notice_if_needed` 删除。一旦这次慢启动超时/失败，标记残留 → 下次启动 `has_update_complete_marker` 仍为真 → **又**走 300s 全量慢路径，陷进去。
6. **🐛 PID 文件只判「存在」不判「存活」**：`Horosa-Web/start_horosa_local.sh` 旧逻辑 `if [ -f "${PY_PID_FILE}" ] || [ -f "${JAVA_PID_FILE}" ]; then exit 1`。上次崩溃/强退/被 kill 残留的死 pid 文件会**永久误拦截**启动（`实现说明` 早记过此坑），失败后叠加第 5 点。

## 改动

> 性质:**纯 Tauri 外壳(`main.rs`)+ 启动脚本(`start_horosa_local.sh`)改动，无 Java 改动 → 不需要重编 `astrostudyboot.jar`**。

### A. 首启等待提示（展示层，零启动逻辑改动）
`main.rs runtime_bootstrap` 首启分支:`emit_progress(78, "更新后首次恢复启动")` → `emit_indeterminate_progress(78, "更新后首次启动需完整校验，约 30–60 秒，请稍候…")`。launcher 前端（`web/app.js` `setProgress`/`__horosaProgress`）支持 `indeterminate`：进度条转为流动动画、数字显示「接收中」，不再像死机。
顺带把 `emit_ready` 的 overlay 文案从「更新准备完成，正在切回新版本」改为中性的「已准备就绪，正在进入主界面…」（旧文案对**正常启动**是错的，且新文案与 launcher 既有「星阙 已准备完成，正在进入主界面…」一致）。

### B. 提速首启校验路径（保留完整校验，只削开销）
- `start_horosa_local.sh` 轮询与 `trusted_runtime` **解耦**：非 trusted 也用 `poll_interval=0.2s`（原 1s）。就绪判定逻辑（`http_responding` + 签名后端）**没动**。
- `start_horosa_local.sh` warmup **改后台非阻塞**：`( … ) >/dev/null 2>&1 &`，`run end` 不再等预热（外层重定向切断对脚本 stdout/stderr 的继承，否则后台进程持有 Rust `command.output()` 的 pipe 致其不返回）。预热**仍完整执行**。
- `main.rs runtime_bootstrap` `skip_mongo_ping = true`（一律跳过启动期 ping；mongo 可选，日常 trusted 路径本就跳过，仅对齐更新后首启）。
- **完整校验未削弱**：首启 `fast_path_enabled=false`（全量 `runtime_dir_is_usable` 校验 python/java）、`trusted_runtime=false`（脚本端完整 verify）**原样保留**。

### C. 修复复发 / 误拦截（正确性）
- **C①** `main.rs` 新增 `consume_update_complete_marker_into_state`：`runtime_bootstrap` 一开头检测到更新标记即**解析通知文本缓存进 `AppState.pending_update_notice` 并立即删除磁盘标记**（兜底强制删，即便解析失败）；`show_post_update_notice_if_needed` 改为从内存缓存取出弹窗。无论本次首启成功/失败，标记都不残留 → 杜绝「次次都慢」。
- **C②** `start_horosa_local.sh` 新增 `prune_stale_pid_file`：读出 pid 用 `kill -0` 探测，**存活才拦截，死 pid 自动清除并继续**。

## 实测验证（提速后 `trusted=0` 路径）

- **提速**：脚本慢路径 **29s → 14s**；日志顺序 `runtime warmup begin (background)` → `services ready` → `run end` 证明 warmup 已后台化、mongo 已跳过。
- **pid 判存活双向**：伪造死 pid（999999/999998）→ 自动清除并正常启动（rc=0）；真活进程再启动 → 正确拦截（rc=1「horosa runtime is already running」）。
- **功能零降级**（三套发布级脚本，全在提速后路径下跑）:
  - `Horosa_Desktop_Installer/scripts/verify_kentang_runtime_endpoints.py --root http://127.0.0.1:8899`：kentang 17 引擎 + 4 chart smoke 全 `http=200 resultCode=0`。
  - `astrostudyui/scripts/verifyHorosaRuntimeFull.js`：exit 0「major Horosa runtime modules responded with valid shapes」（八字/紫微/六壬/流年/卦/金口/十二宫/节气/历法/givenyear/zodialrelease）。
  - `astrostudyui/scripts/verifyPrimaryDirectionRuntime.js`：exit 0（`/chart` 懒加载主限 + `/predict/pd` 对齐，行数 590/1106 一致）。
- **编译**：`cargo check` exit 0、无 warning；`bash -n start_horosa_local.sh` 通过。
