# 奇门历法修复（v2.1.6）：月柱交节边界 + 置闰超神接气定局

> 首发版本：**v2.1.6 / 2.1.6-runtime1**　日期：2026-05-26
> 范围：奇门 `vendor/kinqimen` —— 月柱按精确交节判定 + 置闰法定局改用超神接气置闰后的节气。
> 面向 Windows 同步：**纯 Python（vendor/kinqimen），随 runtime 包 rsync，不需重编 `astrostudyboot.jar`**。
> 来源：Windows 仓库 issue #4（owner 确认）；根因为底层 `kentang2017/kinqimen`（我们 vendored 的引擎，实测与上游同版）的已知未修缺陷。

---

## 1. 根因与背景

奇门排盘由 Python 引擎 `vendor/kinqimen`（图表服务 astropy 调用）完成。节气交接时刻其实早已用 **sxtwl（寿星天文历）精确**计算（`jieqi.py` 的 `get_jieqi_start_date`/`getJieQiJD`），但两处定局逻辑没用上：

- **Bug A — 月柱日级判定（上游 #53、#9）。** `gangzhi()` 用 sxtwl 的 `getMonthGZ()`，它是**日级**——交节当日整日已跳入新月。故交节当日、交节时刻**之前**的时间被错算进新月。例：2005-05-05 立夏交节 17:52，16:30 出生月柱被算成辛巳，应为庚辰。
- **Bug B — 置闰法未做超神接气校正（上游 #62、#43）。** `qimen_ju_name_zhirun()` 旧实现用一堆**农历月/日启发式**判断该用哪个节气定局，约 **45%** 日期误用「当日历法节气」而非「超神接气置闰后的节气」。例：2027-10-31 应为立冬上元（超神），却标为霜降。上游 #43（置闰法与拆补法结果相同）即此类，维护者 2024 声称已修但我们 vendored 的版本仍是旧逻辑。

> 局数表（`jieqicode_jq`）、阴阳遁、三元（`findyuen`，由日柱符头定上/中/下元）经核对**均正确**，与古典转盘文档一致；bug 只在**符头↔节气的配对**与**月柱交节时刻**。

---

## 2. 修复内容

### A. 月柱交节边界（`jieqi.py` `gangzhi`）
取 sxtwl 日级月柱后增加校正：若当日所交为 **12「节」**（立春/驚蟄/清明/立夏/芒種/小暑/立秋/白露/寒露/立冬/大雪/小寒，12「气」不换月柱）之一，且给定时刻**早于**该节精确交节时刻 → 沿用**前一日**的月柱；**立春**同时沿用前一日年柱（年柱边界）。其余日不变（`year < 1900` 走原农历路径，不受影响）。

### B. 置闰超神接气定局（`jieqi.py` `zhirun_jieqi` + `config.py`）
新增 `jieqi.zhirun_jieqi(y,mo,d,h,mi)`，按古典**置闰转盘**返回定局所用节气：
1. **锚点**：当日所属「至」——取其上元符头 `F*`（甲子/甲午/己卯/己酉）≤ 当日的最近一个冬至/夏至（先看下一至的超神前窗，再回溯）。
2. **节气序列**：自该至起按 阳遁[冬至…芒種] / 阴遁[夏至…大雪] 顺序，每个上元符头组（15 天=三元）配一个节气。
3. **置闰**：该至与其上元符头之距 `dgap ≥ 9` 天 → 在 `F*` 组重复一次「大雪/芒種」三元（接气），至之上元顺延至 `F*+15`。
4. 当日自 `F*` 起第 `n=⌊(当日−F*)/5⌋` 元，组 `g=n//3` → 节气 `seq[g]`。

`config.qimen_ju_name_zhirun` 改为：`zhirun_jieqi` 定节气 → 阴阳遁（yydun）+ 局数（`jieqicode_jq`）+ 三元（`findyuen`，符头定，与拆补一致）。新增 `config.dingju_jieqi(...,option)`：拆补(1)=历法节气 `jq`，置闰(2)=`zhirun_jieqi`。

### C. 节气标签（`kinqimen.py` `pan`）
`pan()` 的「節氣」字段由 `config.jq` 改为 `config.dingju_jieqi(...,option)`——置闰盘显示超神后的真实定局节气（如立冬），与排局一致。拆补盘仍显示历法节气。

> 拆补法（`qimen_ju_name_chaibu`）**未改**：实测 534 个日期与古典定义（历法节气+符头元）0 偏差，本就正确。

## 3. 明确「不做」（审计上游全部 issue 后）
- **中宫寄宫**（#56/#54/#27/#23）：值符落中5/天禽寄坤时天盘无中宫，属**转盘奇门寄宫惯例**非崩溃；#27/#26 报告的「天盘返回 tuple 畸形」我们版本**已不复现**（天盘为干净 dict）。本轮不动。
- **神煞类**（#44 六仪击刑/入墓/伏吟/天显时格/五不遇时/宫迫、#12 暗干暗神、#6 五不遇时）：功能**增强**，非 bug，不入本版。
- **安装/用法/环境**（#52/#51/#50/#48/#46/#45/#28/#55）：针对 pip/streamlit/`app.py`，我们是 vendor + 服务方式，**不适用**。
- **同类「日级」隐患**也存在于 `vendor/kinwuzhao`、`kinastro` 的 bazi —— 本轮**不动，仅记录为后续**（避免扩面）。

## 4. 验证

### 4.1 单元测试（`vendor/kinqimen/test_qimen_calendar.py`，11/11 通过）
`python3 test_qimen_calendar.py`：月柱边界（2005-05-05→庚辰、2021-05-05→壬辰、交节后不回归、立春年柱+月柱）；置闰 #62（2027-10-31→立冬上元六局 + 标签立冬）、拆补不变（霜降五局）、#43（拆补处暑一局 / 置闰白露九局）、置闰闰（2027-12-15→闰大雪四局）；2018–2030 `zhirun_jieqi` 节气恒有效、整盘双法不崩。

### 4.2 对照与自洽
- **基准点**：上游 #62（立冬上元六局）、#43（白露上元九局）——新算法全中。
- **手工核验超神漂移**（2004 早春）：上元符头每节气领先约 4–5 天，2004-01-16 甲午（上元符头）领先大寒 5 天 → 大寒上元（旧版误作小寒）；2004-02-01 → 立春上元。与用户提供的权威转盘文档一致。
- **置闰自洽**（2018–2030）：至−符头距呈干净周期（~1→15 后置闰复位），无异常、无崩溃。
- **服务层**：`websrv/webqimensrv.py` `_build_sections` 透出校正后的「节气/排局」（#62 → 排局陰遁六局上元、节气立冬）。
