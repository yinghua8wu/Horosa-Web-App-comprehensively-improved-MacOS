# 更新后无法自动打开（卡页面、需重启两三次）修复 — v2.3.2

## 症状
用户从 2.3.0 → 2.3.1 软件内更新后，**更新装好但不会自动进主界面**，卡在启动器页面；
必须「点击更新 → 重启 → 再次重启」才能打开。2.3.1 已做过「更新后启动卡顿」修复，但症状仍在。

## 真因（实测日志复盘，非推测）

设备 `~/Library/Application Support/com.horacedong.horosa/`：

**1. 自动重开本身没坏。** `logs/update-installer.log` 最近两次更新均：
```
[app] old process exited / [app] install succeeded / [open] attempt 1 / [open] relaunch confirmed / update helper success
```
helper（以 root 运行，因 `/Applications` 改写需管理员）`open_app` 已确认把新版拉起。

**2. 卡的是「更新后首次启动」的全量慢校验。** `logs/horosa-run-issues.log` 显示一次更新后**三次启动**：

| 启动 | 时刻 | trusted_runtime | 就绪耗时 |
|---|---|---|---|
| 自动重开 | 18:23:42 | **0（全量慢路径）** | 21s（冷启动） |
| 手动重启①| 18:26:25 | **0（仍全量）** | 7s |
| 手动重启②| 18:28:12 | **1（快路径）** | 7s → 终于打开 |

**3. 为什么前两次都 trusted=0、第三次才 1：** `runtime_bootstrap`（`src-tauri/src/main.rs`）里
`fast_path_enabled = !first_launch_after_update && … && runtime_fast_path_allowed(...)` —— 更新后首启被
**强制**走全量；而让后续变快的 `.runtime-fast-path.json` 标记**只在「整轮启动成功」后（旧 5510 行）才写**。
冷首启 ~22s 像卡死 → 用户强退 → 标记没写 → 下次 `first_launch_after_update` 虽已 false，但标记仍缺 →
`runtime_fast_path_allowed=false` → **又是全量慢路径** → 反复，直到某一轮被完整跑完才写标记、随后才快。

**4. 这层全量校验其实冗余：** `apply_update` 在安装前已对 runtime 包做
`verify_sha256(&…, plan.runtime_sha256)`（main.rs ~4953）——装进去的 runtime 是「按官方清单逐字节验签过」的，
首启再 300s 全量纯属安全戏法，代价却是这个反复卡顿。

## 修法（`src-tauri/src/main.rs` `runtime_bootstrap`）
对「更新后首启」且 runtime 已验签的情形：**在 `start_runtime` 之前**就给刚装好的 runtime 预写 fast-path 标记，
并取消 `fast_path_enabled` 对 `first_launch_after_update` 的强制关闭：
```rust
if first_launch_after_update {
    if let Some(manifest) = read_runtime_manifest(&paths) {
        write_runtime_fast_path_marker(&paths.runtime_dir, &manifest); // 已验签 → 预写
    }
}
let fast_path_enabled = !force_runtime_install
    && runtime_fast_path_allowed(&paths.runtime_dir); // 不再 && !first_launch_after_update
```
效果：① 更新后首启直接走 trusted 快路径（~7s，对比冷全量 22s+）；② 标记在 warmup/导航**之前**就落盘 →
即便把冷首启当卡死强退，下次仍是快路径，**根除「强退→又全量→重启两三次」**。
兜底未删：若快路径在新 runtime 上失败，原 `first_launch_after_update` 分支会 `clear_runtime_fast_path_marker`
并自动全量复检一次。首启提示文案同步从「需完整校验 30–60 秒」改为快路径版「正在恢复启动，通常约 10 秒」。

## 生效范围（重要）
更新后「首启」跑的是**新版**代码，故本修复进 **2.3.2**：从 2.3.1 → 2.3.2 起，每次更新后首启即快、自动打开。
用户已发生的 2.3.0 → 2.3.1 那一跳由旧版逻辑驱动、无法回溯修，但本修复杜绝其再现。

## 验证
- `cargo check`（src-tauri）EXIT 0、编译通过。
- `release_preflight.sh` `[9] D` 哨兵（grep 注释 sentinel「需重启两三次」）+ `A` 文案哨兵已跟改为「正在恢复启动」。
- 需真机实测：2.3.1 装好本修复后发 2.3.2 → 软件内更新 → 应**自动进主界面、零手动重启**。

## 是否需重编 jar
否。纯 Tauri 外壳（main.rs）改动，不涉 `astrostudysrv`。需 `build_desktop_release.sh` 重新构建+签名+公证桌面包。

## Windows
本修复是 **macOS 专属**（helper 为 bash、`/Applications`、osascript、root 改写都是 mac 语义）。Windows 更新流不同，
若 Windows 有相同「更新后首启反复」症状，按同一思路（已验签 runtime 首启即信任、标记预写、强退不回退全量）改 Windows updater，代码另写。见 `docs/windows-sync-handoff.md`。
