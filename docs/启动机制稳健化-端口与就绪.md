# 启动机制稳健化：根治「端口被占用 / 后端未启动」

> 分支 `feature/startup-port-readiness-hardening`（off `main`）。发布版本号待与在途 `feature/astro-progression-v250`（已占 v2.5.0）协调后定。
> 计划全文见 `~/.claude/plans/port-win-mac-issues-close-polished-gizmo.md`（v4·最保守版，经两轮对抗式红队复核）。

## 背景：两个反复出现的症状
1. **端口被占用 → 软件用不了。**
2. **打开后显示「后端未启动 / 本地排盘服务未就绪」。**

## 根因（逐条对源码证实）
**端口被占用**
- `choose_free_port()`(`main.rs`) 绑 `:0` 拿端口后**立即 drop 释放**再交子进程绑 → TOCTOU 竞态；端口被瞬时抢走后启动脚本 `port_listening` 命中即 `exit 1`，普通路径一次失败即 `return Err` → 用不了。
- 上次崩溃/强退残留的**自家**后端还活着 → 脚本 `runtime_already_live` 直接 `exit 1`。
- 关键再认知:**跨路径/其它随机口的旧孤儿不会导致新启动端口被占**(新启动用 `choose_free_port` 避开已占端口、选随机口),故**无需**到处 `pgrep`/`kill` 的「大反应器」(那会误杀更新换版旧实例/dev 进程)。真正会卡的只有「端口被瞬时抢走」和「自己上次后端卡死」两类。

**后端未启动**
- 就绪探测只验 `/common/time`(轻接口),但 Spring 懒加载开着、warmup 改后台且普通启动跳过 → **第一次真排盘打到冷 bean** → 慢/报错一次,前端 `request.js` **无重试** → 立即弹「未就绪」。
- 7 个引擎中 dunjia/taiyi/jinkou/qizheng 的**主调用是裸 `fetch(buildKentangEndpoint…)`**(非 `request()`),首个排盘冷失败无任何兜底重试。
- 运行中后端异常退出后无看门狗/重连,整局后续全报「未就绪」。
- 已排除伪因:Redis/Mongo **不**阻塞启动(`AstroStudyProgram.java:26-27` 已 exclude;Java 树无 Redis 客户端)。

## 设计原则：最保守 / 零干扰已有功能
> **铁律:每个改动都是「增量式」,只在「今天本来就会坏」的失败/边界路径上触发;启动成功、排盘成功的正常路径与今天逐字节一致。** 凡需改动正在工作的热路径(静态服务器生命周期、`#10` 拦截器/签名、SSE、导航、端口解析、渲染、排盘算法)的方案一律不采用。

## 六个增量式修法

|---|---|---|---|

| 2 | 脚本对端口冲突给可区分退出码(pre-check 占用→`exit 3`;子进程启动期死亡时 grep 子进程日志仅 `Address already in use\|Errno 48\|BindException\|Port [0-9]+ was already in use`→`exit 3`,否则维持 `exit 1`) | `start_horosa_local.sh` | 默认仍 `exit 1`(今天行为);新 grep 全部 `if grep -q`/`\|\| true` 守卫(`set -euo pipefail` 安全);**绝不用裸 `port`**(否则 Spring banner/`--server.port=` 会误判) |

## 为「最稳」主动砍掉的(v3→v4)
- ❌ 单实例插件 / `LSMultipleInstancesProhibited`:会破坏 `open -na` 更新换版(红队 M1);Mac 双开本就由 LaunchServices 兜底。
- ❌ 重写 `command.output()` 为 spawn+timeout:重写正在工作的调用是最高回归风险,「永久卡死」非用户所报症状。
- ❌ 改端口发现机制 / `window.__HOROSA_PORTS__` 注入 / 动 `deriveFromPagePort`/`resolveKentangServiceRoot`:要碰静态服务器+前端解析,且端口发现脆弱性非用户所报症状(查询参数 `?srv=` 本就首选且工作正常)。
- ❌ 到处 `pgrep`/`kill` 的「大反应器」:经推演不必要(跨路径孤儿不致端口被占),且会误杀(红队 H5/M2)。
- ❌ 阻塞式启动遮罩、主动心跳定时器、上调 `-Xmx`(无 OOM 证据)。

## 残余风险(据实列名)
1. Mac 跨版本「手动」双开(非更新换版、非 dev):靠 LaunchServices,极罕见情形下 reclaim 可能误杀——owner-pid 思路已大幅收窄;本轮 reclaim 只针对自家 PID 文件,无此风险。
2. 永久卡死(就绪轮询之前的极少数 hang):本版不改 `command.output()`,未覆盖;若实测出现再以证据补 `-m`/超时。
3. 端口发现刷新丢参数:非所报症状,未动(查询参数首选工作正常)。
4. **Windows 未落地代码**:见 `windows-sync-handoff.md`,逐项实测后才闭环。
