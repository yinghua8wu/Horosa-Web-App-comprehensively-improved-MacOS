## 1. `(1,0)` 边界四柱跨技法一致性（Java）

**根因**：晚子时时柱起干在 Java 有 **4 条独立路径**，全部只写了 `hour==23 && !after23NewDay && lateZiHourUseNextDay`（日柱守今 + 时柱用次日干 → 日干 `+1`），**漏了对称的 `hour==23 && after23NewDay && !lateZiHourUseNextDay`**（日柱进位次日 + 时柱按今日干 → 日干 `+9` 退一位）。

**路径**（均在 `Horosa-Web/astrostudysrv/astrostudy/.../helper/BaZiHelper.java`）：
- `getTimeColumn(dayrCol, timeziIdx, ...)` 与 `getTimeColumn(dayrCol, birth, ...)` —— 八字/果老(`OnlyFourColumns`→`BaZi`)/`/chart` 的 `fourColumns.time`。
- `getTimeGanziStr(...)` —— 六壬/金口的 `nongli.time`（经 `NongliHelper.getNongLi`）。**这条是六壬/金口显示时柱的真实来源**，第一次只修 `getTimeColumn` 时漏掉它导致 `(1,0)` 仍 庚子。
- `getTimeStartGan(...)` —— `getTimeGanZi` 用，补对称分支保持一致。

**结果**：`(1,0)` 27日23:30 → `壬寅 戊子`，与 `baziLunarLocal.js`(jest 139)/奇门(前端 override)/太乙(Python) 全一致。八字/六壬/金口/果老经运行中重编 jar e2e 验证 4/4。**需重编 `astrostudyboot.jar`**。

## 4. AI 供应商连通性（GitHub mac/win 4 issue）

- **Mac #9（真 bug，已修）**：`AIAnalysisProxyService.buildAnthropicBody` 的 content 块过去复用 Gemini 的 `buildTextPart`（只有 `text`）→ Anthropic `/v1/messages` 报 `messages.content: missing field type`(503)，**对话 + 测试连接都挂**。新增 `buildAnthropicTextPart` 补 `type:"text"`。16 个 AIAnalysis 单测通过。
- **Mac #8（已修）**：未登录/未配置凭据走内置 horosa 代理 → 裸 401「No cookie auth credentials」。前端 `AIAnalysisMain.testProfileChat` catch 解码 URL 编码错误 + 对 401/未登录给可操作中文提示。
- **Win #9（增量修，需 Windows 实测）**：`boundless/HttpClientUtility.getHttpHost` 只读 `http_proxy` 等环境变量（GUI 应用常拿不到）→ 加 `http(s).proxyHost` 系统属性回退。纯增量：无属性时返回 null，行为不变。完整 OS 代理自动探测可由启动器加 `-Djava.net.useSystemProxies=true`（Win 端可选后续）。
- **Win #8（已排查 = 非应用 bug）**：Ollama 走 `/v1`（OpenAI 兼容端点，正确），心跳 + 无流式超时已在；「模型加载后 GPU 空转不出字然后停止」属 Ollama/Windows 环境问题，Mac 无法复现修复，建议 Windows 端核查 GPU/驱动/模型。
