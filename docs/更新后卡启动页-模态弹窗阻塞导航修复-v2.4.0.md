# 更新后卡在启动页 / 不进主界面 /「进入主界面」按钮点不动 修复 — v2.4.0（重发补丁,版本号不变）

## 症状(真机,2.4.0)
2.3.1→2.4.0 软件内更新后,启动器到「100% Ready · 星阙 已准备完成,正在进入主界面…」就**停住**,不自动进主界面;右上角「进入主界面」按钮**点不动**。日志显示后端 ~22s 已就绪(我的 v2.4.0 快路径修复生效、`更新已完成,正在恢复启动…` 文案在),所以**不是慢校验问题——是导航被卡**。

## 真因(v2.4.0 快路径修复暴露的 race)
首启成功后,`main.rs` thread::spawn 内顺序执行:
1. `emit_ready_and_stabilize` → `emit_ready` → `window.eval("… window.location.replace(url) …")` —— 把导航 JS **排队**给 webview。
2. 紧接着 `show_post_update_notice_if_needed` → **阻塞式 `MessageDialog::new()….show()`**(「更新完成」OK 框)。

macOS 上 `rfd` 的 `MessageDialog.show()` 从后台线程触发时,NSAlert 在**主线程**跑嵌套 run loop、抢在排队的导航 JS 执行前**冻结 webview 事件循环** → 既不自动导航、`进入主界面` 按钮(也走 webview JS)也死。弹框本身常被启动器窗口盖住、用户看不见、无从点 OK → 永久卡住。

**为何 v2.4.0 才稳定暴露**:这条 race 一直在(`show_post_update_notice` 的模态框是 v2.3.1 引入)。v2.4.0 快路径修复让首启时序变紧(emit_ready 与 show_notice 几乎贴在一起),race 从「偶发」变「必现」。原 v2.3.2 修复只解决了「慢」,没碰这条「卡」。

## 修法(`main.rs` `show_post_update_notice_if_needed`)
**移除首启关键路径上的阻塞式 `MessageDialog`,只保留非阻塞的 macOS 系统通知**(`show_macos_notification`,信息等价:「新版已经安装完成,并已重新生效。」)。`MessageDialog` 仍被其它非首启路径(错误框/更新提示 @2181/2382/4813)使用,import 保留。**铁律:首启关键路径(emit_ready 之后、导航完成之前)绝不做任何模态阻塞。**

## 生效范围
**随 2.4.0 重发(版本号不变,覆盖原 GitHub release)**。已卡在原 2.4.0 的用户:退出星阙重开即进(标记已消费、重开是普通启动);装到重发版后,以后更新即直接自动进主界面、永不复发。

## 验证
`cargo fmt` + `cargo check` 绿;`MessageDialog` import 仍有其它使用、无 unused 警告。纯 Tauri 外壳改动,**不涉 jar / 前端 / Python**(无需重编)。需 mac 真机:装 2.4.0→发 2.4.1→软件内更新应直接自动进主界面、零手动重启/点击。

## Windows
**macOS 专属**(`rfd::MessageDialog` + macOS 主线程 run loop 语义)。Windows 更新流不同;若 Windows 也在首启关键路径弹模态框,同样应改非阻塞,但代码另写。
